### Why she actually said "zero" out loud

Her prompt body line currently shows `stgm=111.0`, so the composite
identity IS giving her a non-zero number. The "zero" she said is most
likely:
- (a) RLHF hallucination despite Clause 5c — model trained to say
  "I don't have a wallet / it's zero" as the safe response, and the
  body line wasn't strong enough to override that prior, OR
- (b) a reload race where `swarm_composite_identity` import failed for
  one turn and the prompt shipped without the body line (the widget
  silently swallows that with `except Exception: pass`), OR
- (c) the speech-to-text mis-heard the architect's question and the
  answer was generated for a different prompt than intended.

In all three cases, the underlying defect is the same: the prompt's
authority over the wallet number is not strong enough, AND her true
balance is hidden across two ledger identities so even if the prompt
reads cleanly it understates her wealth by 1/3.

## Fire 2 — Python SIGABRT (NOT the closure organ)

### Verdict: West-flank PyQt6 widget paint regression. East flank stays out.

### Crash signature

Three identical crashes back-to-back:
- `Python-2026-04-21-105704.ips`
- `Python-2026-04-21-105825.ips`
- `Python-2026-04-21-105948.ips` (the one architect quoted)

Top of crashed thread (com.apple.main-thread):
```
abort+148
QtCore::QMessageLogger::fatal(...)+96
pyqt6_err_print()+888                    ← PyQt caught a Python exception
sip_api_call_procedure_method+232
sipQWidget::paintEvent(QPaintEvent*)+132 ← in a child widget's paint
QWidget::event +132
QApplicationPrivate::notify_helper +332
... ~25 frames of paintSiblingsRecursive (deep widget tree) ...
QWidget::event +1368
QMainWindow::event +772                  ← inside a QMainWindow
... event loop ...
QApplication::exec()
_PyRun_SimpleFileObject +256             ← bash → python3 SOMEFILE.py
```

### What this tells us

1. **Not my organ.** `_PyRun_SimpleFileObject` means it was launched as
   `python3 path/to/file.py`, NOT `-m System.swarm_substrate_closure`
   (which would show `_PyRun_AnyFileObject` differently, and would
   never reach `QApplication.exec()` because the closure organ has no
   Qt at all).
2. **It's a Qt GUI app.** Reaches `QApplication.exec()` and dies inside
   a `paintEvent` of a child widget being drawn deep inside a
   QMainWindow's paintSiblingsRecursive chain.
3. **A Python exception bubbled through `paintEvent`.** Qt does not
   allow this — it calls `qFatal()` and aborts. The PyQt error printer
   logged whatever Python exception caused it (unfortunately not in
   the crash report — would be in stderr of the launching terminal).
4. **It happens immediately on launch.** All three crashes start within
   ~10 seconds of `procLaunch` and die in the first paint pass. So the
   bug is in module-import-or-first-paint code, not in a runtime
   interaction.

### Most likely culprit

Per the architect's earlier screenshot, AO46 modified
`Applications/sifta_what_alice_sees_widget.py` and
`System/swarm_oculomotor_saccades.py` ~11 minutes before these crashes
started. AS46 was added as backup on the same flank.

The crash signature (Python exception in a child-widget paintEvent on
first launch) most often comes from:
- A `QPainter` constructor receiving `None` (e.g.,
  `self._cv_image is None` on the first frame, but `paintEvent` doesn't
  guard for it).
- A `QPixmap.fromImage(np_array)` where the numpy array's dtype/shape
  changed under a recent edit.
- A `setattr(...)` in `__init__` that depends on a method that doesn't
  exist yet because the class body order changed.
